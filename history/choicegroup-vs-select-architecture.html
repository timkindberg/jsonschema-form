Why choicegroup and select stay separate

They look like the same thing — both let you pick from a fixed set of options. But "a choice" is data, and the data is already unified upstream as facts.choices. control.kind is not about the data — it's about which DOM to draw, and a dropdown and a bank of radios are genuinely different markup. Merging them doesn't remove a branch; it just hides it and throws away the type.

The trap: conflating two independent axes

Semantic · WHAT it is

A choice

One value (or many) drawn from a fixed set of labelled options. This is what enum / oneOf mean. It is framework- and widget-agnostic.

Lives once in facts.choices — derived at parse time, consumed by every choice widget.

Presentation · HOW it's drawn

A render archetype

The DOM element family: one <select>, or N <input>s, or a <textarea>… Different element, different focus model, different adapter code.

Lives in control.kind — chosen by present(), switched on by the adapter.

The unification you're reaching for happens at the second box: facts.choices is the single shared representation of "a choice." select, multiselect, radio and checkboxes all read from it. By the time you reach control.kind, the question is no longer "what is this?" but "what element renders it?" — and that answer legitimately splits in two.

Which space each layer lives in

Consumer-space — you author: schema, resolver, overrides Library-space — Core derives & adapters render the boundary — widget, the shared word

Only two seams cross into your space, and each corresponds to one of the earlier answers. A resolver plugs into present() and speaks in widget names — that's why the name is the ergonomic, open axis. renderNode / a custom adapter plugs in at render and consumes control.kind — that's why the kind is the closed, stable IR. Everything in blue is library-owned and invisible unless you deliberately reach in; the surrounding <form> + submit stays yours (ADR 013).

Same data, genuinely different markup

select / multiselectkind: select 
<!-- one focusable control -->
<select name="color">
  <option value="red">Red</option>
  <option value="green">Green</option>
</select>
  • One element, one tab stop
  • Selection state is the element's value
  • Labelled by a single <label for>
  • Options are children (<option>)
radio / checkboxeskind: choicegroup 
<!-- N focusable controls -->
<div role="radiogroup">
  <label><input type="radio" name="color" value="red">Red</label>
  <label><input type="radio" name="color" value="green">Green</label>
</div>
  • N elements, arrow-key roving focus
  • State is per-input checked, not value
  • Needs a group role + wrapper labelling
  • Every input repeats the shared name

So: fold everything into one choicegroup?

✕ "Only choicegroup"

  • The branch doesn't disappear — the adapter still has to decide "<select> vs N inputs." You just move the if inside one arm and pay for it forever.
  • You lose the discriminated union: TypeScript can no longer tell you a select has attrs.multiple and a group has options[].attrs.checked. One blurry attrs bag.
  • It doesn't even DRY anything — the shared part (facts.choices) was already shared upstream.

✓ Archetype per DOM shape

  • Adapters switch (control.kind) over a closed, tiny set — one arm per real DOM shape, exhaustively type-checked.
  • Growth is cheap where it should be: multiselect and checkboxes shipped as "+1 flag, 0 new adapter branches" (same kind, a multiple boolean).
  • Widget name stays the expressive axis (grows freely); archetype stays the stable axis the renderers depend on.

the one-liner

Unify on the data (that's facts.choices, and it already is) — not on the DOM. A <select> and a bank of radios are different elements with different focus and state models, so they earn different control.kind arms. Collapsing them hides a branch you can't delete and discards the type that keeps the adapters honest.
